Payment Default Risk Model

A binary classification model that predicts the likelihood of payment default on e-commerce transactions, built with Azure ML Designer and AutoML.

2025 · Individual project · ~30,000 rows · 45 columns

Lenders and retailers lose money when they can't tell in advance which customers are likely to default on payment. I built a model to predict that risk from transaction and customer data — with a target of 80% F1-score, balancing false positives against missed risk.

Azure ML Designer AutoML Feature Engineering Classification

How I built it

The dataset was a German-language e-commerce dataset with roughly 45 columns and 30,000 rows. Before any modeling, I spent time on schema review and PII handling — identifying which fields were personally identifiable and making sure they were handled appropriately before being fed into training.

I engineered several features to give the model more signal than the raw columns alone: deriving customer age from date-of-birth fields, creating binary flags for specific behaviors, and building item-count features from order-line data. The data was split 80/20 with stratification to keep the class balance consistent across training and test sets, then run through Azure's AutoML to compare model types and select the strongest performer.

Challenges & what I learned

Getting the feature engineering right took more iteration than the modeling step itself. Binary flags and derived features that seemed obviously useful sometimes barely moved the F1-score, while a few simpler ones — like item count per order — turned out to carry real signal.

What I learned

This project sharpened how I think about the order of operations in a real ML workflow: schema understanding and PII handling have to happen before feature engineering, not after, or you end up reworking your pipeline. It also reinforced that a stratified split matters — an imbalanced target variable can make results look better than they are.
